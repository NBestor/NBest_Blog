## Context

当前系统已经具备博客、快写、评论、点赞、Todo、日历等基础能力，但呈现方式仍偏分散：首页主要承载快写，博客在独立列表页，评论是平铺结构且交互较弱，Todo 和日历需要用户进入独立页面才能查看。用户希望首页变成聚合工作台：博客为主、快写为辅，同时把 Todo 和日历放到更顺手的位置。

评论数据目前通过 `comments.target_type` 和 `comments.target_id` 同时服务博客和快写，但缺少父评论字段和评论点赞状态。点赞记录已经有通用 `like_records(target_type, target_id)`，可以复用为 `target_type = 'comment'`。这能避免新增独立评论点赞表。

## Goals / Non-Goals

**Goals:**

- 博客评论和快写评论统一升级为支持回复、嵌套展示和点赞的线程式评论。
- 评论输入框统一为内嵌按钮布局，按钮位于文本框右下角，默认占位文本为 `请说点什么吧~`。
- 首页改为博客/快写组合信息流：左侧约 3/4 为博客词条，右侧约 1/4 为快写词条。
- 博客和快写词条整体可点击进入详情页。
- 新增快写详情页，承载快写正文、点赞、嵌套评论和回复。
- 博客词条显示标题、标签、分类、简介；简介有字数限制，未填写时从正文自动生成短简介。
- 首页显示登录用户的 Todo 和日历辅助入口：左侧可收起 Todo 引导、右上固定日历框。
- 私有 Todo/日历数据只对登录用户显示，游客不暴露私有数据。

**Non-Goals:**

- 不引入新 UI 框架或评论编辑器库。
- 不实现无限深复杂评论编辑后台；本阶段只做前端递进展示和基础回复。
- 不实现评论举报、删除、折叠、置顶、排序切换或通知。
- 不改变现有文章、快写、Todo、日历的权限归属规则。
- 不把首页做成后台仪表盘；仍保持博客产品的轻量阅读感。

## Decisions

1. 评论树使用 `parent_id` 增量迁移。

   在 `comments` 表增加 `parent_id INTEGER`。顶层评论 `parent_id` 为 `NULL`，回复评论引用父评论 ID。查询评论时仍按 `target_type` 和 `target_id` 拉取当前文章或快写的全部评论，再由 service 层组装为树形结构。

   备选方案是在前端按平铺数据自行找父级，但后端仍必须保证回复目标合法，因此数据层保留 `parent_id` 更稳。

2. 评论点赞复用 `like_records`。

   现有点赞服务已按 `target_type` 和 `target_id` 抽象，评论点赞使用 `target_type = 'comment'`。评论响应中增加 `like_count` 和 `is_liked`。这避免新增表，也保持文章、快写和评论点赞模型一致。

3. 评论回复合法性由后端校验。

   新增评论时允许传 `parent_id`。后端必须检查父评论存在，且父评论属于同一个 `target_type/target_id`。不允许回复另一个文章或快写下的评论，也不允许游客评论或点赞。

4. 评论树接口保持兼容。

   博客评论接口和快写评论接口仍使用现有路径：
   - `GET /articles/{id}/comments`
   - `POST /articles/{id}/comments`
   - `GET /quick-posts/{id}/comments`
   - `POST /quick-posts/{id}/comments`

   请求体新增可选 `parent_id`，响应项新增 `parent_id`、`children`、`like_count`、`is_liked`。旧客户端不传 `parent_id` 时仍创建顶层评论。

5. 快写详情页使用独立路由。

   新增 `/quick-posts/:id` 页面。后端新增 `GET /quick-posts/{id}` 读取快写详情，继续沿用现有可见性规则。首页右侧快写词条整体点击进入详情页，详情页复用评论线程组件。

6. 抽取统一评论线程组件。

   前端新增 `CommentThread` 或等价组件，供 `BlogDetailPage` 和 `QuickPostDetailPage` 使用。组件职责：
   - 渲染递进式评论树。
   - 每条评论显示作者、内容、时间、点赞按钮、回复按钮。
   - 管理“回复某条评论”的输入状态。
   - 使用统一评论输入 UI：浅色背景文本框、右下角内嵌提交按钮、placeholder `请说点什么吧~`。

7. 首页聚合使用并行接口请求。

   首页加载：
   - 博客列表：`GET /articles`
   - 快写列表：`GET /quick-posts`
   - 登录用户额外加载 Todo 摘要：`GET /todos`
   - 登录用户额外加载日历当前月或近期事件：`GET /calendar-events`

   游客只加载公开博客和公开快写，不请求 Todo/日历私有接口。

8. 博客简介使用“手写优先，自动兜底”。

   文章已有 `summary` 字段和编辑页摘要输入。本变更明确摘要字数限制，并在列表展示时使用：
   - 如果 `summary` 非空，展示裁剪后的 `summary`。
   - 如果 `summary` 为空，从正文去除 Markdown 标记后截取开头短文本。

   自动摘要可以先在前端列表展示层生成，也可以在 service 层统一生成。为了博客列表和首页一致，优先放在后端 `article_service` 格式化输出中生成展示摘要。

9. 首页 Todo 和日历入口是辅助入口，不替代详情页。

   Todo 引导使用左侧 fixed 或 sticky 控件，默认展开，点击收起。内容只展示少量未完成待办，例如标题和截止日期，点击可进入 `/todo`。

   日历固定框放在首页右上角，不随页面滚动，展示今日或当前月的简要状态，点击进入 `/calendar`。移动端屏幕较窄时，固定日历框需要降级为页面内顶部卡片，避免遮挡正文。

## Risks / Trade-offs

- [Risk] 评论嵌套过深可能造成移动端横向挤压。→ Mitigation：前端限制每层缩进宽度，并在深层回复使用最大缩进。
- [Risk] 首页同时加载博客、快写、Todo、日历，接口数量增加。→ Mitigation：并行请求，游客不请求私有 Todo/日历，失败时单区块显示错误而不阻断首页。
- [Risk] 固定日历框可能遮挡内容。→ Mitigation：桌面端给首页布局预留右上空间，移动端取消 fixed，改为普通卡片。
- [Risk] 评论树迁移可能影响已有评论。→ Mitigation：`parent_id` 默认为 `NULL`，既有评论全部成为顶层评论。
- [Risk] 词条整体可点击与内部按钮冲突。→ Mitigation：卡片使用可点击容器，内部按钮阻止事件冒泡。

## Migration Plan

1. 数据库迁移：为 `comments` 增加 `parent_id`，保留既有评论为顶层评论。
2. 后端评论服务增加树形组装、回复创建、评论点赞统计。
3. 后端快写增加详情接口，文章/快写评论接口支持 `parent_id` 和点赞状态。
4. 前端抽取评论线程组件，替换博客详情和快写详情评论区。
5. 首页改为博客/快写组合布局，接入 Todo 引导和日历固定框。
6. 补充响应式样式，重点验证移动端评论缩进、固定日历降级和 Todo 收起状态。

Rollback 策略：
- `parent_id` 字段可保留不用，旧平铺评论仍能按顶层显示。
- 若首页聚合布局出问题，可临时恢复旧首页只展示快写。
- 若快写详情页出问题，可保留列表展示并隐藏详情入口。

## Open Questions

- 评论嵌套是否需要限制最大可回复层级？设计默认允许多层展示，但视觉缩进设置最大值。
- 评论点赞是否允许取消点赞？设计默认与文章/快写点赞一致，允许点赞和取消点赞。
- Todo 引导展示几条待办最合适？设计默认展示未完成且最近的 5 条。
- 日历固定框展示“当前月事件”还是“今日/近期事件”？设计默认展示当前月中离今天最近的几条。
