## Context

项目当前已经有登录态、私有路由、SQLite 初始化、用户专属数据表、前端受保护页面和全局布局。`/todo` 路由已经在导航规划中存在，并且登录后才显示，但当前仍是占位页。阶段 10 需要把它落成真正的私密待办清单，并在用户进入网站后弹窗提醒到期任务。

Todo 数据必须强制仅自己可见。提醒仅做页面弹窗，不做短信、邮箱、浏览器系统通知或后台定时任务。

## Goals / Non-Goals

**Goals:**
- 为登录用户提供自己的 Todo CRUD 能力。
- 支持待办标题/内容、分类、截止时间、完成状态。
- 提供未完成到期/即将到期 Todo 的查询接口。
- 在前端 `/todo` 页面完成新增、编辑、删除、完成状态切换。
- 登录用户进入网站后看到待办提醒弹窗。
- 复用现有鉴权、路由守卫、导航过滤和 SQLite 初始化模式。

**Non-Goals:**
- 不实现多人共享待办。
- 不实现管理员查看他人待办。
- 不实现短信、邮箱、浏览器 Push 或服务端定时推送。
- 不实现复杂重复任务、优先级矩阵或看板拖拽。
- 不新增外部依赖，除非实现过程中发现现有能力无法满足基本日期处理。

## Decisions

1. 使用单表 `todos` 存储待办。

   字段建议为：`id`、`user_id`、`title`、`content`、`category`、`due_date`、`is_done`、`create_time`、`update_time`。`due_date` 使用 `YYYY-MM-DD` 字符串，便于 SQLite 比较和前端 `<input type="date">` 直接绑定。

   备选方案：拆分 todo_categories 表。暂不采用，因为阶段 10 只要求分类选择/输入，单字段足够，后续如需分类管理再扩展。

2. 后端以用户维度强制隔离。

   所有 Todo API 都通过 `getCurrentUser` 获取当前用户，只查询或修改 `user_id = current_user.id` 的记录。更新、删除、切换完成状态如果找不到当前用户自己的记录，返回 404。

   备选方案：在服务层返回全部记录后前端过滤。拒绝，因为私密数据不能依赖前端过滤。

3. 提醒接口只返回当前用户未完成且需要提醒的 Todo。

   提醒范围使用“已到期或未来 1 天内到期”的未完成待办，覆盖原规划里的到期/即将到期提醒。接口可命名为 `GET /api/todos/reminders`，前端进入网站后请求一次。

   备选方案：只提醒今天到期。拒绝，因为原规划同时包含“到期/即将到期”。

4. 前端提醒弹窗挂在 `AppLayout`。

   `AppLayout` 已经持有登录态并包裹所有页面。新增 `TodoReminderModal` 组件后，登录用户进入任意页面都会查询提醒；游客不查询。用户关闭弹窗后，本次页面生命周期不再反复弹出。

   备选方案：只在首页提醒。拒绝，因为规划是“进入网站自动弹窗提醒”，不应依赖用户先进首页。

5. `/todo` 页面使用普通表单和列表交互。

   页面提供新增表单、待办列表、完成状态切换、编辑保存、删除。分类可以用文本输入配合已有分类快捷选择，避免在阶段 10 引入分类管理复杂度。

   备选方案：弹窗式复杂编辑器。暂不采用，普通表单更符合现有页面风格，也更容易验收 CRUD。

## Risks / Trade-offs

- [Risk] 日期按本地时区理解，后端只存日期字符串，跨时区访问可能有边界差异。 → Mitigation：当前项目定位为个人使用，按浏览器本地日期选择和服务器日期查询即可；文档中明确 `YYYY-MM-DD` 语义。
- [Risk] 用户每次刷新都可能看到同一提醒。 → Mitigation：弹窗提供关闭按钮；本次会话关闭后不重复显示。持久化“已读提醒”不在本阶段范围。
- [Risk] Todo 分类自由输入会出现同义分类。 → Mitigation：本阶段先保证可用，后续可增加分类管理。
- [Risk] 全局提醒查询失败影响页面体验。 → Mitigation：提醒组件失败时静默降级，不阻塞主页面渲染。

## Migration Plan

1. 在数据库初始化中新增 `todos` 表和 `user_id/is_done/due_date` 相关索引。
2. 新增后端 schema、service、API router，并接入总路由。
3. 新增 `/todo` 页面并替换占位页。
4. 新增全局提醒弹窗组件并挂入布局。
5. 验证 API 权限隔离、CRUD、提醒查询、前端页面操作和弹窗行为。

回滚时可移除前端入口和后端 router；数据库新增表不会影响现有功能。
