## Context

当前实现中，搜索接口返回 `articles + quick_notes`，这是对“快记”的直接实现；但用户现在明确要搜索首页的“快写”，也就是 `quick_posts`。快写有公开、好友、仅自己可见的可见性规则，因此搜索快写时必须复用 `quick_post_service.canViewQuickPost` 和 `listVisibleQuickPosts` 一类的现有逻辑。

首页当前被重排成上方快写主区 + 右侧小月历，博客在下方。用户新的布局要求更明确：博客必须在左侧，快写必须在右侧，日历位于右侧上方，快写在日历下方。响应式压缩时可以减少边距、缩窄区域、压缩日历内容，但不能把模块改成上下堆叠，也不能让待办变成覆盖其他功能的大块框。

按钮视觉也需要从字符或文字转为矢量图标。为避免引入额外依赖，优先用本地内联 SVG 组件来表达放大镜、爱心、箭头等图标。

## Goals / Non-Goals

**Goals:**

- 将搜索目标从博客 + 快记修正为博客 + 快写。
- 搜索结果保持左右两栏：博客结果在左侧，快写结果在右侧。
- 快写搜索复用现有快写可见性规则。
- 首页保持左侧博客、右侧日历 + 快写的相对位置。
- 窄屏时只压缩边距、宽度和日历展示密度，不改变博客/快写/待办左右方位。
- 日历在宽度不足时压缩为“日历”极简入口。
- 待办始终保持在页面最左侧的小引导位置，不变成压住其他模块的整块框。
- 搜索、点赞、日历展开等按钮使用内联 SVG 矢量图标。

**Non-Goals:**

- 不搜索快记、待办或日历。
- 不删除快记页面。
- 不引入大型图标库。
- 不改变博客、快写、评论现有点赞 API。
- 不改变博客和快写的权限可见性规则。

## Decisions

1. 搜索响应字段从 `quick_notes` 改为 `quick_posts`。

   后端 `SearchResponse` 改为：

   - `query`
   - `articles`
   - `quick_posts`

   `quick_posts` 使用现有 `QuickPostResponse` 类型。前端 `SearchPage` 对应改为读取 `response.data.quick_posts` 并显示“快写结果”。上一轮 `quick_note_service.searchQuickNotes` 可以保留但不再由搜索 API 使用，避免扩大删除影响。

2. 快写搜索在 service 层复用可见列表。

   在 `quick_post_service.py` 新增 `searchVisibleQuickPosts(query, currentUserId)`，内部先调用 `listVisibleQuickPosts(currentUserId)`，再按 `content` 做大小写归一后的包含匹配。这样可以直接沿用公开、好友、本人可见逻辑。

   备选方案是在 SQL 中直接 `LIKE` 查 `quick_posts`，但那会重复可见性判断，容易漏掉好友关系规则，所以不选。

3. 搜索页使用不可反转的左右双栏布局。

   搜索页使用 `search-results-grid`，桌面和窄屏都保持两列：左列博客，右列快写。响应式只调整列宽、间距和页面边距；当宽度过窄时允许横向滚动或压缩卡片内容，但不改为上下堆叠。

   这样遵守“博客左、快写右，不要上下关系”的要求。为避免移动端严重挤压，卡片内文字使用换行、截断或更紧凑的元信息。

4. 首页采用三块稳定方位：左侧待办引导、左侧博客主栏、右侧日历/快写栏。

   首页主内容使用一个固定左右关系的容器：

   - 最左侧：Todo 引导，始终是窄条或小抽屉，使用 fixed/absolute 左边缘，不变成页面内大卡片。
   - 左主栏：博客入口和博客列表。
   - 右侧栏：顶部日历，下面快写发布与快写列表。

   宽度缩小时，主容器减少左右外边距，右栏变窄，日历先压缩；但博客不跑到右边，快写不跑到左边，Todo 不变成压住内容的普通框。

5. 日历使用渐进压缩。

   桌面宽度足够时，右侧栏顶部显示完整小月历。宽度不足时，日历容器进入 compact 状态：

   - 只显示“日历”两个字或同等极简按钮。
   - 保留点击进入日历页。
   - 不强行渲染月历网格，避免挤占快写栏。

   是否 compact 由 CSS container/media query 实现，优先用 CSS 控制隐藏月历细节，避免 JS 监听宽度。

6. 图标使用内联 SVG 本地组件。

   新增或扩展图标组件，例如 `Icon.jsx` 或在 `LikeButton.jsx` 中内联 SVG。优先实现：

   - `SearchIcon`
   - `HeartIcon`
   - `ChevronIcon`

   搜索按钮使用放大镜 SVG；点赞按钮根据 `isLiked` 切换 fill/stroke；日历展开按钮使用 chevron SVG 旋转或切换方向。所有图标按钮保留 `aria-label` 或可见文本。

   备选方案是引入 lucide-react，但当前项目还没有图标库依赖；为避免包体和配置变化，本次不新增依赖。

7. 快写发送栏放在右栏快写区内。

   用户提到“博客是在快写发送栏下面显示博客”可能表达的是当前快写发布区不应横压博客上方；设计采用更清晰的分区：快写发送栏与快写列表同属右侧快写栏，博客列表独立留在左侧。这样快写功能整体位于右侧，博客整体位于左侧。

## Risks / Trade-offs

- [Risk] 极窄屏保持左右关系会造成内容很窄。→ Mitigation：减少主布局外边距，压缩日历为“日历”入口，卡片文本换行，并允许必要的横向滚动而不改变左右语义。
- [Risk] 用户对“快写发送栏下面显示博客”的描述可能有歧义。→ Mitigation：实现时保持核心约束“博客左、快写右”，快写发布栏归入右侧快写区域，避免压在博客上方。
- [Risk] 内联 SVG 分散在多个组件中可能重复。→ Mitigation：优先抽出小型图标组件，避免重复路径。
- [Risk] 搜索字段变更可能影响已有前端读取 `quick_notes`。→ Mitigation：同步修改 `SearchPage`，并用 API 验证确认返回 `quick_posts`。

## Migration Plan

1. 后端搜索 schema/API 从 `quick_notes` 切换到 `quick_posts`。
2. 新增快写搜索 service，并复用可见性规则。
3. 前端搜索页改为左右双栏：博客左、快写右。
4. 首页重排为博客左栏、日历+快写右栏，Todo 保持最左小引导。
5. 替换搜索、点赞、日历展开的图标为内联 SVG。
6. 运行 API、lint、build 和浏览器布局验证。

Rollback 策略：
- 搜索 API 可临时兼容保留 `quick_notes` 字段为空，同时使用 `quick_posts`。
- 首页布局如出现严重窄屏问题，可先增加最小宽度和横向滚动，仍不改变左右相对位置。
- SVG 图标可回退到字符显示，不影响业务接口。

## Open Questions

- 极窄屏是否允许整个主内容区横向滚动？设计默认允许在无法继续压缩时横向滚动，以保持左右相对位置。
- 快写发送栏是否必须一直显示在右栏顶部？设计默认是，登录用户右栏顶部日历下方显示发送栏，再显示快写列表。
