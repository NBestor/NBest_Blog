## Context

阶段 10 已经实现 Todo 待办和全局 Todo 提醒弹窗。阶段 11 需要新增日历备忘录：月历视图、生日/纪念日/其他事件、年度重复，以及未来 7 天事件提醒。`/calendar` 路由已经在导航中作为登录用户入口存在，但当前仍是占位页。

日历事件与 Todo 一样属于私密个人数据，必须强制按当前用户隔离。提醒仍然只做页面弹窗，不做短信、邮箱、系统通知或后台推送。

## Goals / Non-Goals

**Goals:**
- 为登录用户提供自己的日历事件 CRUD。
- 支持事件标题、日期、备注、类型和每年重复。
- 提供月视图数据，能标注当前月份内的事件。
- 支持生日/纪念日等年度重复事件跨年显示和提醒。
- 登录用户进入网站后看到未来 7 天日历事件提醒。
- 将 Todo 提醒弹窗演进为统一提醒中心，避免多个全局弹窗互相覆盖。

**Non-Goals:**
- 不实现农历、节假日自动导入或时区选择。
- 不实现多人共享日历。
- 不实现管理员查看他人事件。
- 不实现拖拽改期、日程时间段、重复规则编辑器。
- 不新增外部日历库；月历视图用本地日期计算和现有 React/CSS 实现。

## Decisions

1. 使用单表 `calendar_events` 存储事件。

   字段建议为：`id`、`user_id`、`title`、`event_date`、`event_type`、`note`、`is_yearly`、`create_time`、`update_time`。`event_date` 使用 `YYYY-MM-DD` 字符串，`event_type` 限定为 `birthday`、`anniversary`、`other`。

   备选方案：把生日和纪念日拆成独立表。暂不采用，因为当前行为一致，只是类型不同；单表更容易维护和展示。

2. 后端服务层负责年度重复计算。

   非重复事件按完整日期匹配。年度重复事件保留原始年份，但在月历和提醒查询时按当前查询年份映射到对应月日。提醒查询未来 7 天时，需要处理跨年窗口，例如 12 月 28 日查询到次年 1 月 3 日。

   备选方案：前端自行计算重复事件。拒绝，因为权限和提醒查询都应由后端给出一致结果。

3. API 分为事件 CRUD、月份查询和提醒查询。

   建议接口：
   - `GET /api/calendar-events?month=YYYY-MM` 返回该月可显示事件。
   - `POST /api/calendar-events` 创建事件。
   - `PUT /api/calendar-events/{id}` 更新事件。
   - `DELETE /api/calendar-events/{id}` 删除事件。
   - `GET /api/calendar-events/reminders` 返回未来 7 天提醒事件。

   月份查询返回事件时附带 `display_date`，年度重复事件可显示为当前查询月份对应年份的日期，同时保留原始 `event_date`。

4. 前端 `/calendar` 实现轻量月历视图。

   页面维护当前年月，展示 7 列网格。每个日期格展示日期号和事件标签。页面提供新增/编辑表单，可设置标题、日期、类型、备注、每年重复。点击事件进入编辑状态。

   备选方案：引入 FullCalendar 等日历库。暂不采用，阶段需求是基础月历和提醒，手写轻量视图更符合当前依赖规模。

5. 将 `TodoReminderModal` 演进为统一提醒中心。

   现有组件可重命名或替换为 `ReminderModal`，登录后并行请求 Todo 提醒和日历提醒。弹窗分组展示“待办提醒”和“日历提醒”。游客不请求。关闭后本次页面生命周期不重复弹出。

   备选方案：新增第二个 `CalendarReminderModal`。拒绝，因为两个全局弹窗可能同时出现，交互体验差。

## Risks / Trade-offs

- [Risk] 年度重复事件在闰日（2 月 29 日）跨到非闰年时存在歧义。 → Mitigation：本阶段可将非闰年的 2 月 29 日年度事件映射到 2 月 28 日，并在实现中保持一致。
- [Risk] 月历日期计算容易出现本地时区偏移。 → Mitigation：只使用日期字符串和 `new Date(year, monthIndex, day)` 本地构造，不解析 UTC 日期字符串作为业务日期。
- [Risk] 全局提醒中心请求两个接口，任一接口失败可能影响弹窗。 → Mitigation：单个请求失败时该分组为空，不阻塞另一个分组和页面渲染。
- [Risk] 事件较多时月格拥挤。 → Mitigation：本阶段展示前几条事件标签，完整管理仍在列表/编辑区域完成。

## Migration Plan

1. 在数据库初始化中新增 `calendar_events` 表和 `user_id/event_date/is_yearly` 相关索引。
2. 新增后端 schema、service、API router，并接入总路由。
3. 新增 `/calendar` 页面并替换占位页。
4. 将 Todo 提醒组件演进为统一提醒中心，同时接入日历提醒。
5. 验证事件 CRUD、月视图、年度重复、未来 7 天提醒、游客保护和全局弹窗。

回滚时可移除前端页面、提醒中心中的日历分组和后端 router；数据库新增表不影响既有功能。
