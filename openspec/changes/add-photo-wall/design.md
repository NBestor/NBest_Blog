## Context

项目已经存在照片墙的基础实现：`photos` 表、`/api/photos` 列表/上传/修改可见性/删除接口、头像上传和文章图片上传的照片记录归集，以及 `/photo` 前端页面。当前实现已经覆盖一部分阶段8能力，但仍需要按正式规格收敛：明确权限边界、补齐复制图片链接、强化管理员管理边界、完善验证和交互细节。

照片资源由本地静态目录提供，后端通过 `/static/uploads/...` 暴露。认证沿用现有 JWT 与 `getCurrentUser` / `getOptionalCurrentUser`。可见范围沿用 `public` 与 `self`，其中 `self` 表示仅照片所有者可见。

## Goals / Non-Goals

**Goals:**

- 保留并完善现有照片墙实现，让游客可浏览公开照片，登录用户可同时看到公开照片和自己的私密照片。
- 登录用户可上传个人照片，并设置公开或仅自己可见。
- 照片所有者可修改单张照片可见范围、复制图片链接、预览大图、删除照片。
- 管理员可管理全站照片，至少在后端 service/API 权限判断中具备扩展能力，并在前端根据 `can_manage` 暴露操作。
- 头像和文章插图继续自动写入照片记录，作为个人照片墙来源之一。

**Non-Goals:**

- 不重做图片处理流水线，不引入对象存储、CDN、图片审核或复杂压缩策略。
- 不实现多图批量上传、拖拽排序、相册分组和瀑布流虚拟滚动。
- 不在本变更中重构 Markdown 编辑器上传体验。
- 不做真正的后台管理页面；管理员能力通过同一照片墙接口和 `can_manage` 体现。

## Decisions

1. 继续使用现有 `photos` 表作为照片墙唯一数据源。

   表字段保持轻量：`user_id`、`url`、`source_type`、`visible_type`、`upload_time`。头像、文章插图、手动上传照片通过 `source_type` 区分。这样能直接复用已有头像/文章图片归集逻辑，避免为不同来源建多张表。

   备选方案：按来源拆分 `avatar_photos`、`article_images`、`uploaded_photos`。拒绝，因为照片墙展示和权限逻辑一致，拆表会增加查询和权限判断复杂度。

2. 照片列表接口保持单入口 `GET /api/photos`，根据当前用户自动过滤。

   游客只返回公开照片；登录用户返回公开照片和自己的照片。响应中的 `can_manage` 表示当前用户是否能操作该照片。管理员用户后续可被视为拥有全站管理权限，后端判断应统一收敛在 service 层，而不是让前端猜角色。

   备选方案：拆分 `/photos/public` 和 `/photos/me`。暂不采用，因为当前页面需要混合展示公开照片和自己的私密照片，单入口更贴合页面数据流。

3. 管理操作继续使用资源级接口。

   - `POST /api/photos`：登录用户上传照片。
   - `PUT /api/photos/{id}`：拥有者或管理员修改可见范围。
   - `DELETE /api/photos/{id}`：拥有者或管理员删除照片和本地文件。

   这些接口应返回 404 来隐藏无权访问资源的存在，保持和 Todo/日历模块一致。

4. 图片文件保存继续使用本地静态目录。

   手动上传照片写入 `static/uploads/photos/{user_id}/`，头像和文章插图继续使用各自目录。删除照片记录时，如果 URL 指向本地 `/static/uploads/` 文件，则同步删除文件；非本地 URL 只删除记录。

   备选方案：只删数据库记录，保留文件。拒绝，因为会造成长期孤儿文件堆积。

5. 前端以真实资源管理界面替换静态展示。

   `/photo` 页面继续对游客开放。登录用户看到上传面板和可管理照片的操作区；游客只看公开照片。预览使用弹窗显示大图，并提供关闭、复制链接。复制链接优先使用 `navigator.clipboard.writeText`，失败时给出可手动复制的链接文本或提示。

6. 复制链接使用前端能力，不新增后端接口。

   图片 URL 已在响应中返回，前端可拼接成绝对地址并写入剪贴板。这样不增加后端表面面积。

## Risks / Trade-offs

- [Risk] 当前已有照片墙实现可能与正式规格存在细节偏差。 → Mitigation：任务阶段先做差异检查，只补缺口，避免重复实现。
- [Risk] 管理员角色在前端未充分暴露，直接加全站管理可能影响普通用户体验。 → Mitigation：后端先统一权限判断和 `can_manage`，前端只根据 `can_manage` 渲染操作。
- [Risk] 删除本地文件可能误删非照片墙资源。 → Mitigation：只删除 URL 以 `/static/uploads/` 开头且解析到静态目录下的文件，外链或异常路径只删记录。
- [Risk] 浏览器剪贴板权限可能失败。 → Mitigation：复制失败时显示链接文本和失败提示，不阻塞预览和管理功能。
- [Risk] 大量图片时普通网格渲染性能有限。 → Mitigation：本阶段不做虚拟列表，后续照片量增大再增加分页或懒加载。

## Migration Plan

1. 检查现有 `photos` 表是否满足正式规格；必要时补索引或字段约束，保持向后兼容。
2. 完善后端照片权限判断，统一拥有者/管理员可管理逻辑。
3. 补齐前端复制链接、管理态展示和游客/登录态差异。
4. 验证头像上传、文章图片上传、手动上传都会进入照片墙。
5. 回滚时可移除前端新增交互和后端管理员扩展判断；现有 `photos` 表和照片文件不影响其他功能。

## Open Questions

- 管理员是否需要在本阶段拥有独立入口筛选“全部照片”，还是只通过同一公开照片墙管理可见照片？设计默认不新增独立后台入口。
- 图片列表是否需要分页？设计默认本阶段不分页，后续图片数量变大再扩展。
