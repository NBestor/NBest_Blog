## Context

当前首页已经能同时展示博客、快写、Todo 和日历摘要，但日历入口被实现成随视口滚动保持可见的固定浮层。用户实际想要的是首页内容区右上角的小月历：它有页面内固定坐标，是首页布局的一部分，滚动后自然离开视口，不继续贴住屏幕。

顶部导航目前仍承载较多功能标签。新的产品方向是把首页作为统一入口：博客、待办、日历通过首页模块访问，不再作为顶部常驻标签。顶部导航需要更轻，只保留身份相关、照片墙、管理等必要入口，并增加一个可展开搜索控件。搜索只覆盖博客和快记，不覆盖待办或日历。

点赞按钮当前是文字状态。文章、快写和评论需要统一为空心/红色实心爱心，减少文字噪音，同时保留原有接口行为。

## Goals / Non-Goals

**Goals:**

- 将首页日历从随滚动 fixed 浮层改成页面内右上角小月历。
- 小月历显示当前月日期格，并用颜色圈标记本月有事件的日期。
- 小月历默认只显示月历；点击箭头后以悬浮层展开本月事件列表，不挤压页面布局。
- 本月事件列表中，未完成或未到达事件按日期升序显示；已完成或已过去事件置灰并放在底部。
- 重排首页上方布局，使快写与小月历并列，快写尽量占满左侧可用空间。
- 从顶部导航移除博客、待办、日历等首页化入口，但保留路由和权限保护。
- 新增顶部放大镜搜索入口，悬停或聚焦时展开输入框。
- 搜索结果分区展示博客结果和快记结果。
- 博客按标题、标签、分类和简介搜索；快记按正文搜索。
- 文章、快写和评论点赞显示为空心/红色实心爱心。

**Non-Goals:**

- 不删除博客、待办、日历页面本身。
- 不允许通过搜索进入待办或日历。
- 不改变快记的私有属性；快记搜索只返回当前登录用户自己的快记。
- 不引入全文搜索引擎或外部搜索依赖。
- 不把首页小月历做成事件编辑器；编辑仍在日历页完成。

## Decisions

1. 首页日历使用普通布局定位，不使用 `fixed`。

   小月历放在首页上方网格的右侧列，使用普通文档流或相对定位。页面滚动时它会跟随内容自然滚出视口。这符合“固定在首页右上角坐标，但不随滚动跟上来”的要求。

   备选方案是继续 `position: sticky` 或 `fixed`，但这会让日历在滚动时继续贴住视口，和用户描述相反，因此不用。

2. 日历事件展开层使用小月历容器内的绝对定位浮层。

   小月历容器设置 `position: relative`，事件列表用 `position: absolute` 挂在月历下方或覆盖下半区域，`z-index` 高于首页内容。这样展开事件列表时不会挤压快写、博客或快写栏布局。

   箭头状态由前端本地 state 控制：默认收起，箭头向上；展开后箭头向下。本月事件悬浮层内提供进入日历页的入口。

3. 小月历只展示当前月。

   首页小月历不承担完整日历导航。它根据当前日期计算当月网格，并用 `/calendar-events?month=YYYY-MM` 返回的事件标记日期。点击更完整管理入口进入 `/calendar`。

   备选方案是在首页支持切换月份，但这会让首页小组件变成完整日历页的重复实现，本次不做。

4. 本月事件排序在前端完成。

   现有日历接口已经按月份返回事件。首页将事件分成两组：
   - 活跃事件：日期为今天或未来的事件，按 `display_date` 升序。
   - 已完成/已过去事件：日期早于今天的事件，置灰并放在底部。

   如果未来 Todo 式完成状态被加到日历事件，再把 `is_done` 并入同一排序规则即可。本次先按日期是否已过去判断。

5. 首页上方布局使用“快写主区 + 日历侧区”。

   首页顶部建立独立的 `home-top-grid`：左侧为快写发布和快写流，右侧为小月历。下方再展示博客主列表和其他内容。这样快写能在首页首屏更充分铺开，日历也能保持右上角坐标。

   桌面端建议比例为 `minmax(0, 1fr) 300px` 或接近 3:1；移动端降级为单列，日历显示在快写之后或之前，避免横向挤压。

6. 顶部导航通过配置隐藏入口，不删除路由。

   在 `route-config.js` 为博客、待办、日历设置 `showInNav: false` 或引入更明确的 `homeEntryOnly` 配置。`privatePaths` 仍从 `visibility` 计算，因此隐藏导航不会影响直接访问保护。

   AppLayout 继续使用现有 `canShowRoute` 过滤逻辑，只需要尊重这些配置。博客、待办、日历仍可通过首页按钮、卡片或程序导航进入。

7. 搜索入口放在 AppLayout，搜索页独立路由。

   AppLayout 增加一个搜索表单：默认只有放大镜按钮宽度，hover/focus-within 时展开输入框。提交非空关键词后跳转 `/search?q=...`。

   搜索页 `SearchPage` 读取 query 参数并调用后端统一搜索接口。结果分成“博客结果”和“快记结果”两个区块。游客只会得到博客结果；未登录状态不请求或不展示快记结果。

8. 后端搜索使用统一 `/api/search` 接口。

   新增 `backend/app/api/search.py`，返回结构类似：

   - `articles`: 可见博客结果。
   - `quick_notes`: 当前登录用户快记结果，游客为空。

   搜索逻辑放在 service 层：
   - `article_service.searchVisibleArticles(query, currentUserId)`：复用现有可见性过滤，匹配标题、summary、分类名、标签名，以及后端生成的展示简介。
   - `quick_note_service.searchQuickNotes(userId, query)`：只查当前用户自己的 `quick_notes.content`。

   SQLite 匹配先使用 `LIKE`，并在服务层做大小写归一和去空格。因为规模和需求暂时较轻，不引入 FTS。

9. 点赞爱心抽成轻量组件或统一 CSS 类。

   为文章、快写和评论点赞按钮统一使用相同结构：

   - 未点赞：`♡` 或图标库空心心形，普通颜色。
   - 已点赞：`♥` 或图标库实心心形，红色。

   为减少依赖和改动，优先用文本符号 + CSS；如果项目已有图标库再改为图标组件。按钮仍调用原有点赞/取消点赞接口，disabled 逻辑不变。

## Risks / Trade-offs

- [Risk] 首页小月历事件悬浮层可能遮挡快写内容。→ Mitigation：只在用户主动展开时显示，设置明确最大高度和滚动，并用 `z-index` 控制层级。
- [Risk] 隐藏顶部博客/待办/日历后用户找不到入口。→ Mitigation：首页对应模块保留明确按钮或整卡入口；博客结果也可通过搜索进入。
- [Risk] SQLite `LIKE` 搜索在数据量大时性能有限。→ Mitigation：当前先满足轻量搜索；未来再独立升级 FTS，不阻塞本次需求。
- [Risk] 快记搜索涉及私有内容。→ Mitigation：后端只在已登录时按 `currentUser.id` 查自己的快记，游客返回空快记结果。
- [Risk] 使用字符爱心在不同字体下视觉差异。→ Mitigation：用 CSS 固定尺寸、颜色和行高；必要时后续替换为图标库。

## Migration Plan

1. 新增搜索 API 和 schema，不影响现有文章、快记接口。
2. 前端新增搜索页和导航搜索入口，保留所有现有路由。
3. 调整 `route-config.js` 的导航显示配置，只隐藏顶部标签，不删除路由。
4. 重构首页顶部区域：快写与小月历并列，小月历事件列表改为悬浮展开层。
5. 将点赞按钮逐步替换为统一爱心样式。
6. 运行后端搜索、权限和前端构建验证。

Rollback 策略：
- 搜索入口可从 AppLayout 暂时隐藏，不影响原有功能。
- 导航隐藏配置可恢复为显示博客、待办、日历。
- 小月历组件可退回为普通摘要卡片，不影响日历详情页。

## Open Questions

- 快记搜索结果点击后，是进入快记页并高亮定位，还是先进入快记页并展开对应内容？设计默认进入 `/quick/note`，并可通过 query/hash 定位到对应快记。
- 日历事件“已完成”的语义目前日历数据没有显式字段，设计先按事件日期早于今天视为已过去并置灰。
- 搜索框展开触发是否只用 hover/focus，还是移动端点击放大镜后展开？设计默认桌面 hover/focus，移动端点击后保持展开。
