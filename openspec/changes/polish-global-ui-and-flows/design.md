## Context

项目已经完成多轮功能迭代，前端页面包括首页动态、博客列表/详情/编辑/草稿、个人中心、关注、照片墙、Todo、日历、快记和设置页。整体风格以简约博客式布局为主，主要样式集中在 `frontend/src/styles/global.css`，少量页面 JSX 中存在局部结构差异。

阶段12不是新增业务模块，而是质量收敛：统一视觉层级、减少重复样式、补齐状态呈现，并通过 API 和 in-app browser 对核心流程与权限边界做复测。后端只在复测发现真实缺陷时做最小修复。

## Goals / Non-Goals

**Goals:**

- 统一全站页面的布局节奏、面板样式、按钮层级、表单输入、列表卡片、弹窗和提示信息。
- 保证桌面和移动端关键页面没有明显重叠、溢出、按钮文字挤压或动态内容撑坏布局。
- 为常见状态建立一致表达：加载中、空状态、成功提示、错误提示、权限不可见。
- 复测游客、登录用户、资源拥有者、非拥有者和管理员相关权限边界。
- 修复复测中发现的阻断性 UI、权限或流程 bug。

**Non-Goals:**

- 不引入新的 UI 框架、组件库或图标库。
- 不做品牌重设计，不改变当前简约博客风格。
- 不重写所有页面结构，不做大规模 CSS 架构迁移。
- 不新增业务功能，不改变已确认的业务规则。
- 不将所有页面改造成像后台管理系统一样的高密度布局；保留博客产品的轻量阅读感。

## Decisions

1. 以现有全局 CSS 为收敛中心。

   继续使用 `global.css` 作为主要样式入口，补充通用 class，例如统一的操作区、状态提示、紧凑表单和弹窗结构。已有页面局部 class 仅在必要时调整，避免引入新依赖或 CSS module 迁移。

   备选方案：引入组件库或 Tailwind。拒绝，因为当前项目已有稳定 CSS 基础，引入新体系会放大改动范围。

2. 优先修“跨页面一致性问题”，再修单页细节。

   第一层处理全局规则：按钮高度、输入框焦点态、卡片/面板间距、弹窗 z-index、移动端 grid 收敛。第二层处理具体页面：编辑器、照片墙、Todo、日历、首页动态等页面的特定溢出或状态问题。

3. 保持用户状态驱动的导航规则。

   导航入口继续由路由配置和登录状态控制。质量收敛只验证和修复可见性问题，不改变“能不能进入”的业务规则，也不新增导航分组逻辑。

4. 权限复测采用 API + 浏览器双线。

   后端权限边界使用 API 请求验证，覆盖游客、登录用户、非拥有者和管理员；前端使用 in-app browser 验证用户真实可见内容和操作入口。这样避免只看 DOM 或只看 API 导致遗漏。

5. 修复范围遵循最小改动。

   如果复测发现问题，优先在已有 service、API 或页面中局部修复。只有多个页面出现同类 UI 问题时才抽成通用样式；不借阶段12做功能重构。

## Risks / Trade-offs

- [Risk] 全局 CSS 调整可能影响多个页面。 → Mitigation：每次改动后使用 in-app browser 抽查关键页面，并运行 lint/build。
- [Risk] 质量阶段容易范围膨胀。 → Mitigation：任务清单只允许修复明显 UI/流程/权限缺陷，不新增业务能力。
- [Risk] 浏览器自动化输入在部分场景可能受限。 → Mitigation：优先使用逐键输入和 API 预置测试数据；无法自动验证的交互明确记录。
- [Risk] 管理员场景缺少公开创建入口。 → Mitigation：验证时可在测试脚本中将测试用户角色升级为 admin，不新增生产入口。
- [Risk] 过度统一会削弱页面差异。 → Mitigation：统一基础控件和状态，不强制所有页面使用相同布局密度。

## Migration Plan

1. 盘点关键页面和全局样式现状，确认高风险 UI 点和复测路径。
2. 优先补全通用样式和响应式基础规则。
3. 按模块修复明显的页面状态或交互不一致。
4. 运行 lint/build。
5. 使用 API 验证权限边界，使用 in-app browser 验证关键页面视觉和流程。
6. 回滚时可逐步撤销本变更中的 CSS 和局部 JSX 调整；业务数据结构原则上不变。
